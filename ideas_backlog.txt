Parallel plugin execution (deterministic fork-join within ExecuteTick)
─────────────────────────────────────────────────────────────────

Currently TickScheduler::ExecuteTick enqueues a [](){} sentinel as a
barrier, then runs BeforeTick/Dispatch/Step/Hook (including PluginManager::TickAll)
sequentially in the coordinator thread.  The worker pool is unused for
real work.

Future work:  Replace the sentinel with N real plugin tasks (one per
loaded plugin) submitted to the workers, then barrier on all N completing
before advancing the clock.  The execution pipeline would become:

    determinism_.BeforeTick(tick)
    ═══ fork ─────────────────────────────
        Worker 1: plugin_1->on_tick(tick)
        Worker 2: plugin_2->on_tick(tick)
        Worker 3: plugin_3->on_tick(tick)
        …
    ═══ join (pending_tasks_ == 0) ──────
    event_bus_.Dispatch()
    clock_.Step()
    on_tick_hook_(tick)

Constraints (to preserve determinism):

  • Plugin iteration order must be deterministic (use std::map or
    explicit priority ordering — not unordered_map).
  • The fork-join must be a strict barrier: all plugins finish before
    Dispatch/Step/Hook, no work-stealing across tick boundaries.
  • No external caller may inject tasks into the worker pool during
    the tick (EnqueueTask remains private).
  • Plugin on_tick must be re-entrant safe (or documented as not) —
    multiple plugins running concurrently must not share mutable state
    without synchronization.

Prerequisites (from the core-hardening backlog):

  • Phase 1 — PluginManager thread safety (already done: mutex +
    snapshot iteration prevents data races on the plugins_ map during
    concurrent TickAll).
  • Phase 2 — Condition-variable workers (block instead of polling)
    ensure workers wake promptly when tasks arrive.
  • PluginManager must iterate plugins in a deterministic order
    (currently unordered_map — change to map or ordered container).

Priority: after Phases 1-4 of the core-hardening backlog are
complete.  This is the feature that turns the worker pool from a
synchronization barrier into real parallel execution.
